一、Model（模型层，保存项目数据）

1. Project
功能：
- 整个项目的核心类。
- 管理所有 Task、Dependency、Resource、Assignment。
- 提供增加、删除、查询等操作。
- 负责项目整体的数据管理。

2. Task（抽象基类）
功能：
- 表示一个任务。
- 保存所有任务共有的信息，如名称、工期、开始时间、结束时间等。
- 作为所有任务类型的父类。

3. BasicTask（普通任务）
功能：
- 继承自 Task。
- 表示普通任务，没有额外特殊功能。

4. MilestoneTask（里程碑任务）
功能：
- 继承自 Task。
- 表示里程碑任务，与普通任务区别在于持续时间等规则。

5. Dependency
功能：
- 表示两个任务之间的依赖关系。
- 记录前驱任务和后继任务。
- 用于关键路径计算、拓扑排序等。

6. Resource
功能：
- 表示项目中的资源。
- 可以是人员、设备、材料等。
- 保存资源自身的信息。

7. Assignment
功能：
- 表示 Task 与 Resource 的关联关系。
- 一个 Task 可以对应多个 Resource。
- 一个 Resource 也可以分配给多个 Task。
- 保存一次资源分配的信息（如数量、工时等）。


二、Importer（导入模块）

8. Importer（抽象基类）
功能：
- 定义导入接口。
- 负责将文件读取为 Project 对象。

9. PPMImporter
功能：
- 继承自 Importer。
- 实现 PPM 格式文件的导入。


三、Exporter（导出模块）

10. Exporter（抽象基类）
功能：
- 定义导出接口。
- 负责将 Project 保存到文件。

11. PPMExporter
功能：
- 继承自 Exporter。
- 实现 PPM 格式文件的导出。


四、Controller（控制层）

12. Controller（单例）
功能：
- 接收用户操作。
- 调用 Project 完成业务逻辑。
- 调用 Importer、Exporter 完成文件读写。
- 协调 View 与 Model。


五、View（界面层）

13. ConsoleView
功能：
- 显示菜单。
- 获取用户输入。
- 输出操作结果。
- 不直接修改任何项目数据。

2.软件基本功能要求
2.1项目模型导入。根据给定文件名，将以特定格式编码存储的项目
调度模型文件导入到内存，转化为可编辑、可执行调度计算的可
重用类对象。项目模型文件格式有多种（如微软 Project XML、
Primavera P6 XER 等），要求至少支持 PPM 格式（示例模型文件
为 Sample.ppm；PPM 文件格式说明见附录 1）。说明：要求“至
少支持 PPM 格式”的目的（支持导入多种格式的功能不加分），
是促进类架构设计优化，从可能支持多种模型格式的角度，需要
考虑导入器（Importer）类的继承体系，而非单独写一个为本作
业专用的 PPM 导入器。
2.2项目模型导出。将内存中的项目模型对象，根据给定文件名，以
特定格式编码写入项目文件。要求至少支持 PPM 格式。说明：要
求“至少支持 PPM 格式”的目的（导出多种格式的功能不加分），
是促进类架构设计优化，从可能支持多种模型格式角度出发，则
需要考虑导出器（Exporter）类的继承体系，而非单独写一个为
本作业专用的 PPM 导出器。
2.3人工项目模型修改
2.3.1 列出所有 Task 对象，按容器索引顺序显示其序号、名称、
工期、前置任务列表和后继任务列表。显示格式自拟。
2.3.2 删除指定 Task 对象。通过容器索引定位并删除指定 Task。
删除时，必须同时删除与此 Task 相关联的所有 Dependency（无
论该任务是作为前置还是后置），且该任务占用的资源引用关系
自动解除；
2.3.3 通过给定名称、工期，向项目中添加一个 Task。名称不可
与已有任务重复；若工期为 0，则自动创建为里程碑任务
（MilestoneTask），否则创建为普通任务（BasicTask）。
2.3.4 列 出 指 定 Task （ 通 过 容 器 索 引 ） 的 所 有 后 继 Task
（Successors）和前驱 Task（Predecessors），显示其容器索引
和名称。显示格式自拟。
2.3.5 修改指定 Task 对象。修改指定 Task 的名称或工期。若修
改 后 工 期 变 为 0 ， 则 Task 类 型 自 动 转 为 里 程 碑 任 务
（MilestoneTask）；若工期大于 0，则自动转为普通任务
（BasicTask）。注：此为 Task 修改模式，修改前后任务在容器
中的索引保持不变。
2.3.6 列出所有 Dependency 对象。显示每个依赖的序号、前置任
务的容器索引、后置任务容器索引。
2.3.7 删除指定 Dependency 对象。根据容器索引或前/后置任务
信息删除指定的依赖关系。
2.3.8 添加 Dependency 对象。通过给定前置任务、后置任务、依
赖类型和 Lag，向项目中添加一个非重复的 Dependency（即不能
存在两对前置、后置完全相同的依赖）。添加后必须保证整个项
目依赖图不产生循环依赖环（DAG 约束）。
2.3.9 列出所有 Resource 对象。显示每个资源的容器索引、名称、
单位成本。
2.3.10 添加 Resource 对象。通过给定名称和单位成本，添加一
个 Resource。名称不可重复。
2.3.11 为 Task 分配 Resource。通过任务序号和资源序号，为指
定 Task 分配一个 Resource。一个 Task 可分配多个 Resource，一
个 Resource 可被多个 Task 同时使用。
2.4显示统计信息：显示当前项目模型中的 Task 总数、Dependency
总数、Resource 总数，以及当前项目的总工期（即关键路径长度，
单位为天）。显示格式自拟。
2.5项目模型验证与调度计算。
2.5.1 验证 Project 合理性（IsValid）。需同时满足以下条件，
才判定为合理：① 依赖图无环（严格有向无环图 DAG）；② 所
有 Task 均位于至少一条从某起始节点到某终止节点的路径上（不
存在孤立或不可达的悬挂节点）；③ 所有 Dependency 中引用的
前后置 Task 均真实存在。
2.5.2 执行关键路径调度计算（Inference）。在 Project 具有合
理性（IsValid == true）的前提下，执行关键路径法（CPM）计
算：① 默认所有无前置依赖的任务（入度为 0）的最早开始时间
（ES）为第 0 天；② 通过逐层前向传播，计算每个 Task 的最早
开始时间（ES）和最早完成时间（EF）；③ 从终止任务反向传播，
计算每个 Task 的最晚开始时间（LS）和最晚完成时间（LF）；④
输出项目的总工期（即所有任务中最大的 EF 值）；⑤ 输出位于
关键路径上的所有任务 ID 列表（按拓扑顺序）。凡满足 EF == LF
或 ES == LS 的任务，即为关键路径上的任务。
3.设计与实现要求
3.1除程序主函数（广义的主函数）、用于运算符重载的友元函数、
必要的 lambda 表达式外，不允许出现任何一个非类成员函数。
3.2不可使用非类成员的全局变量（文件作用域或全局作用域）。
3.3不可将任何受访问规则限制的数据成员作为公有成员。
3.4任何不改变对象状态（不改写自身对象数据成员值）的成员函数
均需显示标注 const。
3.5全部类分为三大类：界面类（开发环境提供的、与图形/非图形交
互界面相关的类， MVC 模式中的 V）、业务流程/控制器类（用于
和界面实现耦合，可以认为是没有界面的整个软件功能集合，MVC
中的 C）、可重用类（包括但不限于支撑本软件底层功能实现的
类，特别重要之处在于：可重用类不是仅为本软件独特需求设计，
而是可以脱离大作业特定要求的、尽可能便利的、在其他应用中
被重用的类，MVC 中的 M）。（此条为强烈建议，未实现界面类和
可重用类的解耦将严重影响成绩）
举例：Task、Dependency、Project 等都可脱离大作业具体功能
要求，在工程和任务管理领域独立大量运用，其类体系和封装设
计不能仅仅考虑做大作业“刚好够用”。
3.6界面类必须以类的形式存在，不可直接写为 main 函数，或其他不
属于类非静态成员的函数。
3.7仅有界面类可以用开发环境自动生成代码框架。（图形界面不加
分，非图形界面不减分）
3.8仅在业务流程/控制器类可以使用开发环境提供的数据类型和函
数。比如，用 QT 做图形化界面，控制器类可接收来自界面的 QT
独有的 QString 类型输入，而 Model3D 的 Name 属性则不可使用
QString 类型。
3.9业务流程/控制器类，特别限定：在不被修改代码的前提下，任何
情况都只能产生 1 个对象实例（设计时需使用单例模式）。
3.10 可重用类只允许使用 C++11 及之后标准支持的标准语法、标准
模板库。不依赖于具体编辑器特性，可以跨开发环境、编译环境、
操作系统编译运行。
3.11 不可在自己编写的代码（一定包含业务流程/控制器类、可重
用类，可能包含界面类）中直接调用操作系统 API。
3.12 任何第三方库（非 C++标准提供、非操作系统提供、非开发环
境提供）的使用，只能处于源代码级别，不可依赖 lib/so/dylib
文件等（静态库也不可以）和 DLL 文件。全部第三方库/代码，来
自网络示例代码及改编代码，均需标注来源和版权信息，且需要
根据编码规范调整。
3.13 大作业是 OOP 的大作业不是 C++的大作业，需要综合运用课上
知识和技能，侧重于以优雅的设计，实现多个、可被更多场合、
更应用便捷使用的类。一切以“我为人人”为出发点，而不是为
了实现大作业功能“刚好够用”。
